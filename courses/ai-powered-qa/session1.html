Session 1: AI Foundations for QA

Shifting the Mindset

14th July 2026 • 2 Hours • Theory + Practical

🎯 Learning Objectives

  • Distinguish between AI hype and practical AI-driven testing capabilities
  • Identify AI tools and agent capabilities relevant to QA engineering
  • Explain spec-driven development and its role in AI-augmented QA
  • Articulate how the QA role is evolving — and the guardrails needed
  • Critically evaluate AI-generated test artifacts

📝 Key Concepts

  • Three Levels of AI: Assisted (human drives) → Augmented (AI handles routine, human validates) → Autonomous (AI plans & executes)
  • Tools vs Agents: A tool is a single transaction. An agent has goals, memory, and multi-step planning.
  • Spec-Driven Development: Clear specs = clear AI output. Vague specs = garbage output.
  • The Loop: Spec → AI Code → AI Tests → Validate → Refine (continuous)
  • Guardrail Zones: Green (automate freely) → Yellow (human reviews) → Red (human only)

📋 Exercise: Spec Template

APP NAME: _______________
PURPOSE: One sentence — what does it do?
TARGET USER: Who is this for?

USER STORIES (3–5):
As a [user], I want to [action], so that [benefit]

ACCEPTANCE CRITERIA (per story):
Given [context]
When [action]
Then [expected result]

EDGE CASES:
- What happens when [boundary condition]?
- What happens when [error condition]?

NON-FUNCTIONAL:
- Performance expectations
- Browser/device support

✍️ Writing Testable Specs — 5 Principles

  • 1. Given/When/Then — behavioral expectations in structured format
  • 2. Explicit boundaries — "1–100 characters" not "enters text"
  • 3. Negative cases — what should NOT happen
  • 4. Data constraints — valid formats, ranges, types
  • 5. Error behaviors — what happens when things go wrong

🔑 Key Stats to Remember
