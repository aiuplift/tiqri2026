Not every team is ready for the same AI integration level. Let's figure out where you are.

QA Maturity Spectrum

Level 1
ManualAd-hoc testing with no automation framework. Tests are executed by hand, results tracked in spreadsheets. High effort, low repeatability.
Ad-hoc testing, no automation, tribal knowledge
Level 2
AutomatedTest frameworks in place (Cypress, Playwright, Selenium), running in CI/CD. Repeatable and consistent, but tests still designed and maintained entirely by humans.
Framework in place, CI integration, stable suites
Level 3
AI-AugmentedAI assists human QEs: generating test cases, suggesting assertions, prioritizing regression. Humans still review and approve. Where most teams should aim first.
AI assists test creation, smart prioritization, co-pilot usage
Level 4
AI-NativeAI plans test strategies, executes autonomously, self-heals broken tests, and predicts quality risks. Human sets goals and guardrails, AI handles execution.
Self-healing tests, autonomous agents, predictive quality
Where is your team today?
Most teams sit between Level 2 and early Level 3. The gap between levels represents not just tooling changes but cultural and process shifts.
Poll: Where do you think TIQRI sits? Most teams are between Level 2 and early Level 3. That's okay - it means the opportunity is huge.

LevelAdopt NowAI capabilities that match your current maturity level. Low risk, high probability of success, measurable ROI within one sprint.NOT YetCapabilities requiring foundations you don't have yet. Trying these too early leads to frustration and the conclusion that 'AI doesn't work for us.'

Level 3 — Augmented Agent workflows, predictive analytics, coverage optimization Removing all human review from critical paths

Building Guardrails
for AI in QE Practice
Using AI effectively, not recklessly
The most strategically important topic. Guardrails separate successful adoption from dangerous over-reliance.

The Guardrail Framework

✅ AUTOMATEAI can run freely with periodic audit. Low-risk, repetitive tasks: test data generation, environment setup, log parsing. Mistakes are cheap to fix.
AI runs freely, periodic audit
• Test data generation
• Regression suite maintenance
• Flaky test detection
• Log analysis & categorization
• Environment health checks
• Test report generation
• Selector updates
⚠️ GATEAI proposes, human approves. Medium-risk creative work: new test cases, defect triage, coverage analysis. One bad output could waste significant time.
AI proposes, human approves
• New test case creation
• Defect triage suggestions
• Coverage gap analysis
• Test strategy changes
• Priority assignments
• Risk assessments
• Performance baselines
🚫 PROHIBITHumans only, AI excluded entirely. High-risk decisions: security sign-offs, GDPR compliance, production releases. Mistakes have legal/contractual consequences.
Humans only, AI excluded
• Security test sign-off
• Compliance verification
• Production release decisions
• Client data handling
• Regulatory assertions
• Incident root cause (final)
• Contract-bound test evidence
Revisit from Session 1 with more depth. They will define their OWN zones in the practical. Emphasize: this is not about limiting AI - it is about using it effectively.

Acceptance Criteria for AI Artifacts

Artifact TypeAccept When...Reject When...
Generated Test Maps to requirement, has assertion, passes review, no hallucinated selectors No clear requirement traced, flaky, untestable assertion
Test Data Valid schema, realistic distribution, no PII, covers edge cases Contains real user data, invalid format, single happy-path only
Defect Report Reproducible steps, clear expected/actual, environment noted Hallucinated steps, wrong environment, duplicate not caught
Code Suggestion Follows team patterns, has error handling, passes lint + tests Security vulnerabilities, deprecated APIs, no error handling
💡 Teams should collaboratively define these criteria. They become your AI quality contract.
Concrete. For each type of AI output, what is the bar? Teams should define this together.

Accountability Model

Who owns AI-generated coverage?
The QE who reviewed and committed it. Same as any code.
Who is responsible if AI tests miss a bug?
The team that accepted the coverage as sufficient. AI is a tool, not a colleague.
Can we blame AI for defect escapes?
No. You cannot delegate accountability to a tool. If you shipped it, you own it.
🔑 Key Principle: "If you can't explain what it validates, don't commit it."
Important organizational clarity. AI is a tool - humans are accountable. The reviewer who approves owns it.

Data & Compliance Framework

Data TypeCan Share with AI?Conditions
Internal Code✅ Yes (enterprise plan)Enterprise license, data not used for training
Test Data (synthetic)✅ YesVerified no PII leakage, anonymized
Client Specifications⚠️ ConditionalCheck NDA, use enterprise tools only, no public AI
Production Data❌ NeverUse synthetic equivalents only
Defect Reports⚠️ ConditionalSanitize client names, redact sensitive URLs
Attribution Model
• AI-generated code: mark in commit message
• AI-assisted tests: note in test header
• Review ownership: same as manual code
• License: team owns all outputs
Compliance Considerations
• GDPR: No personal data to AI tools
• SOC2: Document AI usage in processes
• ISO 27001: Include AI in risk register
• Client contracts: Review AI clauses
