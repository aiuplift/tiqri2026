Spec-Driven Development
The foundation for AI-augmented workflows
This is perhaps the most important concept in the entire series. Emphasize: the quality of your specification determines everything downstream — the quality of the code, the tests, and the AI output. Bad input → bad output, amplified 10x with AI.

Why Specs Matter More in an AI World

⚡ The quality of your specification directly determines the quality of your AI output
❌ Without Clear Specs
"The login should work properly"
↓ Vague input to AI
↓ Generic, shallow tests
↓ False confidence
↓ Bugs reach production
😰 Coverage theater
→ REWRITE AS SPEC
✅ With Clear Specs
"Given valid credentials,
When user submits login form,
Then redirect to dashboard within 2s"
↓ Precise AI input
↓ Boundary + negative cases
↓ Genuine coverage
↓ Defects caught early
🛡️ Real confidence
💡 Live Demo: Paste a vague requirement → see bad output. Rewrite as spec → see the difference.
LIVE DEMO OPPORTUNITY: Paste a vague requirement into an AI tool, show mediocre output. Then rewrite as a proper spec with Given/When/Then, show dramatically better output. This 2–3 minute demo is worth more than any slide. The visual difference is immediate and convincing.

Writing Testable Specifications

5 principles that make AI output dramatically better

📐 Given / When / Then
"Given valid credentials, When submit, Then redirect to dashboard"
📏 Explicit Boundaries
"1–100 characters" not "user enters text"
🚫 Negative Cases
"Empty email → show error, NOT proceed silently"
🔢 Data Constraints
"Price: 0.01–99999.99, 2 decimal places, USD only"
💥 Error Behaviors
"3 failed logins → lock account 15min → show 'Account locked, try again at [time]'"
❌ Untestable
"Users can log in"
→
✅ Testable
"Given valid email + 8-char password, When submit, Then redirect within 2s. Max 3 attempts → 15min lockout."
Practical tips they'll apply directly in the exercise. The spec template in the handout follows these exact principles. Encourage them to look critically at their own past user stories against these five criteria. Show a quick example: bad spec vs good spec side by side.

Guardrails for AI in QE
Trust = Capability × Predictability × Boundaries
Final theory topic. This sets up the accountability mindset they carry into the practical session and beyond. AI without guardrails is like automation without assertions — it runs, it looks impressive, but you can't trust the results.

The Three Zones of AI Trust

Where to let AI run free, where to review, and where to draw the line

🟢 GREEN ZONE
Automate FreelyAI can do this without human review. Low risk, repetitive tasks where mistakes are easily caught and reversed. Example: test data generation, code formatting.
✓  Test data generation
✓  Code formatting
✓  Report generation
✓  Environment setup
✓  Log parsing
🟡 YELLOW ZONE
Human ReviewsAI generates output, but a human must validate before it enters the pipeline. Medium risk, where mistakes could affect quality. Example: test case generation, bug triage.
⚠️  Test case generation
⚠️  Code suggestions
⚠️  Bug classification
⚠️  Test prioritization
⚠️  Coverage analysis
🔴 RED ZONE
Human OnlyAI must not be used here. High risk, compliance-sensitive, or decisions requiring accountability. Example: security sign-offs, production release decisions.
🚫  Security decisions
🚫  Compliance sign-off
🚫  Risk assessment
🚫  Release approval
🚫  GDPR data handling
