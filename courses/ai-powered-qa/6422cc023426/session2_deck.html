02
Prompt Engineering for QA
This is THE core skill
This is THE core skill. A good prompt is the difference between useful output and garbage. Everything you learn here, you'll use in the practical session immediately.

Anatomy of a Good Prompt

ContextWho is the AI pretending to be? What domain expertise does it need? Example: 'You are a senior QA engineer specializing in financial systems.'
Set the role, domain, and background
Spec / InputThe actual requirement or acceptance criteria you want tested. The more specific, the better the output. Always use Given/When/Then format.
Provide the requirement or acceptance criteria
ConstraintsBoundaries the AI must work within: which framework (Playwright/Cypress), which patterns (POM), which languages (TypeScript), what NOT to include.
Define boundaries: framework, style, coverage targets
FormatWhat structure should the output have? Code? Table? Numbered list? Specifying format prevents the AI from guessing and gives consistent, usable results.
Specify output structure: code, table, list
ExamplesShow the AI one good example of what you want. Few-shot examples dramatically improve output quality — the AI mimics the style and depth of your example.
Show one good example of expected output
Every prompt you write should include most of these layers
Show the structure. Every prompt you write should have most of these components. The better your input, the better the output. This is the foundation — techniques come next.

Techniques That Work

TECHNIQUEQA EXAMPLE
Role PromptingAssign the AI a specific expert persona. 'You are a senior SDET with 10 years Playwright experience' sets the quality bar higher than a generic request. "You are a senior SDET with 10 years Playwright experience..."
Few-shot ExamplesProvide 1-2 examples of the exact output format you want. The AI pattern-matches against your examples, producing more consistent results. Provide 1-2 example test cases before asking for more
Chain-of-ThoughtAsk the AI to reason step by step: 'First analyze the spec, then identify scenarios, then write tests.' Forces structured thinking instead of jumping to code. "First analyze the spec, then identify edge cases, then write tests"
Constraint AnchoringExplicitly state technical boundaries: 'Use Playwright, TypeScript, Page Object Model, data-testid selectors only.' Prevents the AI from using wrong frameworks. "Use Page Object Model, max 1 assertion per test"
Negative PromptingTell the AI what NOT to do: 'Do NOT use hard-coded waits, do NOT generate generic assertions.' Eliminates common AI bad habits. "Do NOT use hard-coded waits or sleep statements"

Playwright + AI

Why Playwright + AI?
• Auto-wait eliminates flaky tests
• Locator strategy = readable selectors
• AI models well-trained on PW syntax
• Built-in assertions are expressive
• Trace viewer for debugging
• Multi-browser from single codebase
// Traceability naming test('[Story-1] should display login form', async ({ page }) => { await page.goto('/login'); await expect( page.getByRole('heading') ).toHaveText('Sign In'); await expect( page.getByLabel('Email') ).toBeVisible(); });
Traceability: test('[Story-ID] should...') links tests back to requirements
Quick Playwright recap for those less familiar. AI tools are well-trained on Playwright syntax — it's the best-supported framework for AI-assisted test generation. Traceability naming links every test to a spec item.

Evaluating AI Output &
Setting Guardrails
AI will confidently generate incorrect output — you need the skills to evaluate critically
AI will confidently generate incorrect output. You need the skills to evaluate critically. This section builds your review muscle.

5-Dimension Evaluation Framework

DIMENSIONKEY QUESTION🚩 RED FLAG
CoverageDoes the test suite cover ALL acceptance criteria from the spec? If your spec has 5 ACs and only 3 have tests, coverage is incomplete regardless of test count. Are all acceptance criteria tested? Happy path only
CorrectnessDo the assertions actually check the RIGHT thing? A test that asserts a button exists is not the same as asserting clicking it performs the correct action. Do assertions validate the RIGHT thing? Trivial checks
Completeness Edge cases, boundaries, negatives? Missing error paths
Maintainability POM, readable, no duplication? Hard-coded values
TrustworthinessCan this test actually FAIL when the feature is broken? If a test can never fail (tautology, phantom selector, try/catch swallowing errors), it provides zero value. Can this test actually fail? Always-pass logic
📋 Use the Evaluation Checklist handout for every AI-generated test
This is your review framework. For every AI-generated test, ask these 5 questions. Reference the evaluation checklist handout — you'll use this in the practical session.

Spotting AI Hallucinations

🚨 Trivial Assertions
expect(true).toBe(true)
🚨 TautologiesComparing a value to itself: expect(x).toBe(x). This can NEVER fail because x always equals x. It looks like an assertion but validates nothing.
expect(title).toBe(title)
🚨 Missing Negatives
No error path tests at all
🚨 Phantom SelectorsLocators targeting elements that don't exist in the actual DOM: page.locator('.made-up-class'). The AI invents selectors that look plausible but aren't real.
page.locator('.non-existent-class')
🚨 Tests That Cannot Fail
No real assertion or try/catch swallows
⚠️ Impressive-looking code ≠ correct code
These are the most common AI test generation failures. Show real examples from actual AI output. Key message: impressive-looking does not equal correct. Always verify assertions manually.

Guardrail Principles

1
Never commit AI-generated code without human review
2
Assertion audit: verify every expect() checks something meaningful
3
Negative test requirement: every feature needs at least one failure test
4
Traceability requirement: every test links to a spec requirement
5
Manual exploration first: understand the feature before generating tests

Common Pitfalls
Quick warnings before we go hands-on
Quick warnings before we go hands-on. Each pitfall is drawn from real-world adoption failures.

Pitfalls to Avoid

⚠️ Over-reliance Syndrome
Accepting AI output without critical review leads to fragile test suites. 40 tests generated, but 30 test the same happy path.
⚠️ Hallucinated Coverage
Tests exist but validate nothing meaningful — false sense of security. Green pipeline, zero real assertions.
⚠️ False Confidence
Green CI pipeline masks gaps that only manual exploration would find. Professional-looking output discourages questioning.
⚠️ Copy-Paste Trap
Duplicated AI patterns without understanding create maintenance nightmares. Treat AI output as a draft, never the final version.
