## Task 2: Build AI Adoption Roadmap — Model Answer

### 2.1 Process Map (Example: Warehouse Team)

```
Requirements → Sprint Planning → Development → Code Review → QA Testing → Deployment
     ⚠️              ⚠️                                           ⚠️          
  Ambiguous       No risk-based      ✅ Copilot        ⚠️ Manual    6-hour
  stories         prioritization      in DoD          regression   pipeline
```

Pain points marked:
- ⚠️ Requirements: stories sometimes ambiguous, edge cases missed
- ⚠️ Planning: no AI-driven test prioritization 
- ⚠️ QA Testing: 6-hour regression suite, mostly manual selection
- ⚠️ Release: long pipeline blocks deployment

### 2.2 Effort vs. Impact Matrix:

```
HIGH IMPACT │                          │
            │  AI Test Prioritization   │  Custom AI Agent for
            │  (Quick Win - DO FIRST)   │  Config-Based Testing
            │                          │  (Strategic - PLAN)
────────────┼──────────────────────────┼──────────────────────
            │  AI Spec Review          │  Self-Healing Tests
            │  (Nice-to-Have)          │  (Avoid for now)
LOW IMPACT  │                          │
            │                          │
            └────── LOW EFFORT ────────┴──── HIGH EFFORT ──────
```

### Top 3 Opportunities (Prioritized):

**1. AI-Assisted Test Prioritization (Quick Win)**
- What AI does: Analyzes code diff → recommends which tests to run → cuts regression from 6 hours to <2 hours
- Guardrails: Human reviews AI's "skip" suggestions before execution. Full suite runs nightly as safety net.
- Success metric: Regression time reduced by 60%+ within 30 days
- Effort: Low (integrate with existing CI)
- Impact: High (unblocks daily deployments)

**2. AI Spec Review for Testability (Quick Win)**
- What AI does: Scans user stories in sprint backlog, flags ambiguous language and missing acceptance criteria
- Guardrails: AI suggestions reviewed by PO + QA before sprint commitment. AI doesn't modify stories directly.
- Success metric: 50% reduction in "story unclear" defects within 60 days
- Effort: Low (prompt templates + manual workflow)
- Impact: Medium (catches issues earlier = cheaper to fix)

**3. Configuration-Based Test Generation (Strategic)**
- What AI does: Given 400 customers × 500 parameters, AI generates test scenarios for critical parameter combinations
- Guardrails: Generated tests reviewed by domain expert. Coverage focused on high-risk configurations first.
- Success metric: Configuration-related defects reduced by 30% within 90 days
- Effort: High (needs custom tooling, domain knowledge encoding)
- Impact: High (addresses core business complexity)

### 2.3 Guardrails Definition (for Opportunity #1):

| Zone | Activities |
|------|-----------|
| **Automate Freely** | AI suggests which tests to run based on code diff. AI generates test priority scores. |
| **Gate (Human Reviews)** | Human approves "skip" list before execution. QA lead signs off on reduced suite. |
| **Prohibit** | AI cannot skip security tests. AI cannot skip smoke tests. AI cannot modify test code. |

**Accountability:** QA lead owns the decision to trust AI's prioritization. If a skipped test would have caught a production bug, the team reviews and tightens the "never skip" list.

**Review process:** Weekly retro on AI prioritization accuracy. Monthly metrics: did we miss any bugs that a skipped test would have caught?
