⚡ Prompt Engineering — Anatomy of a Good QA Prompt

┌─────────────────────────────────────────────┐
│ CONTEXT     │ Role, domain, project context │
│ SPEC/INPUT  │ User story, acceptance criteria│
│ CONSTRAINTS │ Tech stack, patterns, limits   │
│ FORMAT      │ Expected output structure       │
│ EXAMPLES    │ What good output looks like     │
└─────────────────────────────────────────────┘

🛠️ Prompt Template: Test Case Generation

Role: You are a senior QA engineer specializing in [domain].
Context: We're testing [feature] in [app description].
Spec: [paste user story + acceptance criteria]
Tech stack: Playwright with TypeScript
Patterns: Page Object Model, AAA (Arrange-Act-Assert)

Generate comprehensive test scenarios including:
- Positive flows (happy path)
- Negative flows (invalid inputs, unauthorized access)
- Boundary conditions (min/max values, empty states)
- Edge cases (concurrent actions, timing issues)

For each scenario, provide:
1. Scenario name
2. Preconditions
3. Steps
4. Expected result
5. Priority (Critical/High/Medium/Low)

🛠️ Prompt Template: Spec Review

Review this user story for testability issues:
[paste story]

Identify:
1. Ambiguous language that could be interpreted multiple ways
2. Missing acceptance criteria
3. Untestable requirements
4. Missing edge cases and boundary conditions
5. Implicit assumptions that should be explicit

For each issue, suggest a specific improvement.

🔍 5-Dimension Evaluation Framework

  • Coverage: Does this test what the spec requires?
  • Correctness: Are assertions checking the right things?
  • Completeness: Are edge cases and negatives covered?
  • Maintainability: Can a human read and update this?
  • Trustworthiness: Would you bet on this in production?

⚠️ Spotting AI Hallucinations

  • Tests that check elements exist but not their content
  • Assertions that match expected result to itself (tautologies)
  • Tests with no meaningful assertions (just expect(page).toBeVisible())
  • Coverage that looks comprehensive but tests nothing meaningful
  • 20 tests for login — but none verify authentication actually works

🚧 Guardrail Principles

  • Never commit unreviewed AI-generated code
  • Every test needs at least one meaningful business logic assertion
  • For every positive test → at least one negative/boundary test
  • Every test must link to a spec item (traceability)
  • Manual exploration first — AI generates from spec, exploration finds what spec missed
